# Клиенты 4phone

## MicroSIP для Windows

Каталог `microsip` содержит дорабатываемые исходники MicroSIP.
Клиент собирается в голосовом варианте без видеокодеков.
Первая целевая сборка: `Release | Win32`, она работает и на 64-битной Windows.

Исходная версия:

- MicroSIP 3.22.3
- источник: https://www.microsip.org/source
- исходный архив: `MicroSIP-3.22.3-src.7z`, не хранится в репозитории
- SHA-256: `fd00f8c6d302d4a98825595c63880d23f986cbaca3652410cc4971d460740e98`
- лицензия: GNU GPL v2 или более поздняя версия, как указано в исходных файлах

На странице загрузок может быть более новая готовая сборка, но 3.22.3 является
последней версией, для которой автор опубликовал архив исходников на момент
начала этой доработки.

## Реализованная схема входа

1. Пользователь вводит email и пароль от 4phone.
2. Клиент вызывает `POST /api/auth/login`.
3. Если включена 2FA, клиент запрашивает код или резервный код и вызывает
   `POST /api/auth/2fa/verify-login`.
4. Клиент получает только активные добавочные, назначенные текущему
   пользователю, через `GET /api/extensions/softphone/assigned`.
5. Один добавочный выбирается автоматически. При нескольких добавочных
   показывается список.
6. SIP-настройки загружаются через
   `GET /api/extensions/softphone/config/:extensionId` и автоматически
   применяются к MicroSIP.
7. Refresh token хранится в Windows Credential Manager. SIP-пароль в INI
   защищается Windows DPAPI для текущего пользователя.

Сохраненный SIP-аккаунт используется только при сетевой ошибке или временной
ошибке API уровня 5xx. Ошибка авторизации, отзыв назначения или пустой список
добавочных требуют нового входа. Выход отзывает refresh token на сервере,
ставит локальный признак выхода и удаляет учетные данные Windows.
При смене пользователя или отключении добавочного backend автоматически
ротирует SIP-пароль и сбрасывает действующую регистрацию FreeSWITCH.
Неуспешный сброс остается в очереди до восстановления ESL. После запуска
backend один раз очищает все регистрации внутреннего профиля, поэтому
перезапуск процесса не оставляет старые регистрации действующими.

## Языки интерфейса

Клиент поддерживает русский, узбекский на латинице и английский. При первом
запуске язык выбирается по языку интерфейса Windows: русский и узбекский
распознаются явно, для остальных языков используется английский. Выбор можно
изменить в `Settings` в поле `Interface language`. Новый язык применяется
после перезапуска 4phone.

Английские строки встроены в исходный интерфейс. Русский и узбекский пакеты
копируются рядом с EXE при сборке. Скрипт `build/generate-langpacks.py`
обновляет их из официального переводчика MicroSIP и добавляет строки 4phone.

## Сборка

Официальный архив содержит только код приложения. Он не содержит точную
сборочную копию PJSIP и бинарные зависимости из каталогов `pjlib`, `pjsip`,
`pjmedia`, `pjnath`, `third_party` и `lib`, на которые ссылается
`microsip.vcxproj`.

Скрипт `build/build-windows.ps1` готовит зависимости и собирает клиент. Для
запуска нужны Windows, Git и Visual Studio 2022 с компонентами:

- MSBuild
- Desktop development with C++
- C++ MFC for the latest v143 build tools
- Windows 10 или 11 SDK

Запуск из PowerShell:

```powershell
cd clients
.\build\build-windows.ps1
```

Скрипт использует:

- pjproject 2.17, commit `5a457451fa2712ba18e12b01738e8ff3af2b26fd`
- защитный патч PJSIP
  `d6a0e7f76611c3a6f530ee051e3e7a622bb1748c`
- проверку результата патча по blob
  `cdc99ff9ed5ba4d6ef2ba8c5740ff23f58c3c86c`
- Opus, pugixml и WinSparkle через зафиксированный baseline vcpkg
- Windows Schannel для TLS без поставки OpenSSL
- обязательную проверку сертификата SIP-сервера

Результат: `microsip/Release/4phone.exe`, `WinSparkle.dll`,
`langpack_russian.txt` и `langpack_uzbek.txt`.

### Установщик Windows

Скрипт `build/build-installer.ps1` собирает установщик Inno Setup после
основной сборки клиента. Для локального запуска нужен Inno Setup 6 или новее:

```powershell
.\build\build-windows.ps1
.\build\build-installer.ps1
```

Результат: `microsip/Release/4phone-setup.exe` и файл контрольной суммы
`4phone-setup.exe.sha256`.

Установщик:

- устанавливает клиент для текущего пользователя в
  `%LOCALAPPDATA%\Programs\4phone` без запроса прав администратора;
- автоматически создает ярлыки в меню «Пуск» и на рабочем столе;
- регистрирует установленный режим клиента, поэтому настройки хранятся в
  `%APPDATA%\4phone`, а локальные файлы и логи в `%LOCALAPPDATA%\4phone`;
- переносит `4phone.ini` и `Contacts.xml` из каталога установщика или с
  рабочего стола, если в профиле еще нет этих файлов;
- запускает 4phone после завершения интерактивной установки;
- добавляет стандартное удаление через список установленных приложений
  Windows, сохраняя пользовательские настройки.

### GitHub Actions

Workflow `.github/workflows/build-windows.yml` проверяет pull request и
изменения `main`, собирает Release Win32 на runner `windows-2022` и публикует
временный CI artifact. Сторонние GitHub Actions закреплены по полному commit
SHA.

Workflow `.github/workflows/release-windows.yml` запускается только по тегу.
Beta выпускается как явно неподписанный prerelease. Stable требует отдельные
SignPath-подписи EXE и installer с timestamp, затем Ed25519-подпись
WinSparkle, SPDX SBOM, SHA-256 и GitHub provenance. Закрепленные версии и
хеши release-инструментов находятся в `build/release-config.json`. Полная
процедура описана в `RELEASE.md`.

Backend пока выдает транспорт `udp` по умолчанию. Перед переключением
`SOFTPHONE_SIP_TRANSPORT=tls` нужно продлить сертификат `sip.4phone.uz`.
Проверка 13 июля 2026 года показала, что сертификат истек 8 июля, а
`certbot.service` завис в DNS hook с 8 июня.

Клиент использует собственный appcast
`https://api.4phone.uz/api/client-updates/windows/appcast.xml`, встроенный
Ed25519 public key и пользовательское подтверждение установки. Он не
обращается к серверу обновлений MicroSIP.
